# Suppliers Module: Technical Documentation & Design Parity

This document outlines the database design, business logic, and API specifications for the **Suppliers Module**. It is designed to ensure 100% parity between the **SQL Server (Backend)** and **SQLite (Frontend/Local)** to support the Online/Offline synchronization strategy.

## 1. Database Design (Data Parity)

All tables must implement the **BaseEntity** structure to support synchronization. IDs must be **GUIDs (uniqueidentifier)** generated at the point of creation (Local-First).

### 1.1 Shared Metadata (Base Schema)
Every table in the Suppliers module must include these fields:
*   `CreatedAt` (DateTime): UTC timestamp of record creation.
*   `UpdatedAt` (DateTime, Nullable): UTC timestamp of the last update.
*   `IsDeleted` (Boolean): Soft-delete flag (records are never physically deleted).

### 1.2 Table: Suppliers
Stores the master data for all vendors/suppliers.

| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `SupplierId` | GUID | Primary Key | Unique identifier generated by the app. |
| `SupplierName` | String | Required, Unique | Name of the supplier (e.g., "Main Meat Provider"). |
| `Phone` | String | Optional | Contact phone number. |
| `Address` | String | Optional | Physical address. |
| `Notes` | String | Optional | General remarks. |
| `CurrentBalance` | Decimal | Default: 0 | Total debt owed to the supplier (increases with invoices, decreases with payments). |
| `IsActive` | Boolean | Default: true | Status flag to disable a supplier without deleting history. |

### 1.3 Table: SystemPaymentMethods
Stores global payment channels available in the system (e.g., bank accounts or InstaPay handles).

| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `SystemPaymentMethodId` | GUID | Primary Key | Unique identifier. |
| `MethodType` | String | Required | Enum-like string: `InstaPay`, `VodafoneCash`, `BankAccount`, `Other`, `Cash`. |
| `AccountData` | String | Optional | Details (e.g., IBAN, Mobile Number, or Handle). |
| `Notes` | String | Optional | General remarks about this payment method. |
| `IsActive` | Boolean | Default: true | Whether this method can currently be used for payments. |

### 1.4 Table: SupplierPayments
Records all financial transactions (outbound money) paid to the supplier.

| Attribute | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `SupplierPaymentId` | GUID | Primary Key | Unique identifier. |
| `SupplierId` | GUID | Foreign Key | Reference to `Suppliers.SupplierId`. |
| `PurchaseInvoiceId` | GUID | Foreign Key (Nullable) | Reference to a specific `PurchaseInvoice` (if debt-settling). |
| `PaymentDate` | DateTime | Required | When the payment occurred. |
| `Amount` | Decimal | Required, > 0 | Amount paid. |
| `SystemPaymentMethodId` | GUID | Foreign Key | The method used (ref: `SystemPaymentMethods`). |
| `CreatedByUserId` | GUID | Foreign Key | The user who recorded the payment. |
| `DeviceId` | GUID | Optional | The device from which the payment was recorded. |
| `Notes` | String | Optional | Remarks. |

## 2. Business Logic & Validations

The logic must be identical in the **WPF Local Service** and the **ASP.NET Core Server Service**.

### 2.1 Supplier Management
*   **Unique Names**: Before creating/updating, verify the name doesn't exist (case-insensitive).
*   **Opening Balance**: When creating a supplier, an `OpeningBalance` can be provided which initializes the `CurrentBalance`.
*   **Balance Integrity**: `CurrentBalance` is a calculated snapshot but acts as a source of truth for the local UI. It must be updated atomically with transactions.

*   **Active Check**: Payments can only be recorded using methods where `IsActive = true`.

### 2.3 Payment Processing (Critical Logic)
When a payment is recorded, the following logical steps must occur:
1.  **Amount Validation**: Must be greater than zero.
2.  **Payment Scenarios**:
    *   **Payment against a specific Purchase Invoice** (`PurchaseInvoiceId` has a value):
        *   Must verify invoice exists and belongs to the supplier.
        *   Invoice must not be fully paid (`RemainingAmount > 0` and `Status != "Paid"`).
        *   Payment `Amount` must be $\le$ `Invoice.RemainingAmount`.
        *   Subtract `Amount` from `Invoice.RemainingAmount` and add to `Invoice.PaidAmount`.
        *   Update `Invoice.Status` to `Paid` (if remaining is 0) or `PartiallyPaid`.
        *   Subtract `Amount` from `Supplier.CurrentBalance`.
    *   **Payment directly to Supplier Account** (`PurchaseInvoiceId` is null):
        *   Supplier must have an outstanding balance (`CurrentBalance > 0`).
        *   Payment `Amount` must be $\le$ `Supplier.CurrentBalance`.
        *   Subtract `Amount` from `Supplier.CurrentBalance`.

### 2.4 External Impacts (Purchasing Module)
While not in the Suppliers table itself, the **Purchase Invoice Approval** logic directly impacts this module:
*   Upon **Approving** a Purchase Invoice:
    *   The `Invoice.RemainingAmount` is added to `Supplier.CurrentBalance` (increasing the debt).

### 2.5 Payment Method Modification & Deletion
*   **Modification**: When updating a payment method:
    *   Update `MethodType`, `AccountData`, and `IsActive` status.
    *   Update the `UpdatedAt` timestamp.
    *   This logic must be applied locally to the SQLite database to maintain parity.
*   **Deletion (Soft Delete)**:
    *   Set `IsDeleted = true`.
    *   Update the `UpdatedAt` timestamp.
    *   Soft-deleted methods should be excluded from the UI and payment processing.

## 3. Synchronization & Offline Strategy

### 3.1 Local-First Operations
1.  **Write**: The user creates a Supplier or Payment. It is saved immediately to SQLite.
2.  **Outbox**: A record is added to the `SyncOutbox` containing the Operation Type (Create/Update), the Entity Name, and the GUID.
3.  **Background Sync**: When internet is detected:
    *   **Upload**: Pending records in `SyncOutbox` are sent to the Server API.
    *   **Conflict Resolution**: Since GUIDs are unique and the system uses `UpdatedAt` timestamps, the server generally adopts the client's state for new records. For updates, the server may check if the server's `UpdatedAt` is newer than the client's version.

### 3.2 State Consistency
*   **Soft Deletes**: Always filter for `IsDeleted = false` in all queries.
*   **UTC Consistency**: All timestamps (`CreatedAt`, `PaymentDate`, etc.) must be stored in **UTC** to avoid timezone mismatches during sync.

## 4. API Endpoints (For WPF Client Implementation)

The WPF app should use these endpoints when Online:

*   `GET /api/Supplier`: Fetch all active suppliers.
*   `POST /api/Supplier`: Create a new supplier (Admin Only).
*   `GET /api/Supplier/{id}/payments`: History of payments for one supplier.
*   `POST /api/Purchase/payment`: Record a new payment (Updates balance and invoices).
*   `GET /api/SystemPaymentMethod`: List available global payment methods.
*   `POST /api/SystemPaymentMethod`: Add a new global payment method (Admin Only).
*   `HttpPut /api/SystemPaymentMethod/{id}`: Update an existing global payment method.
*   `HttpDelete /api/SystemPaymentMethod/{id}`: Soft-delete a global payment method.
*   `HttpPatch /api/SystemPaymentMethod/{id}/toggle`: Toggle active status.
